## 🆕 신규 슬라이드 (사례집 통합)

1. **테스트 파트너: 김실장의 자동사냥** — 개발자 잡담 채널을 녹음 대상으로 활용해 장시간 샘플 확보 + 용어집 시드 구축(게임명 자동 추출 → 소울류·카잔·브라운 더스트·메탑·리세마라 등 18개). 비개발자에 실용적인 "실전 환경이 없으면 성격이 비슷한 공개 콘텐츠를 스파링 파트너로" 메시지.
2. **AI 리뷰로도 못 잡는 것** — Codex 적대적 리뷰가 race/재시도/rate-limit은 전부 잡았는데, "재시도 버튼이 완료 전사 4개를 삭제"하는 UX 구멍은 실제 사용에서 발견 (세그먼트 파이프라이닝 PR1 경험). "코드를 직접 못 봐도 한 번이라도 써보는 것이 가장 강력한 검증" 메시지.
3. **운영하면서 또 배운 3가지** — PWA는 언제든 죽는다(iPad 제스처 134세그먼트 손실) · AI도 관성에 빠진다(Flash 회귀, 유의어 드리프트) · 로그는 1시간 만에 사라진다(Vercel 무료 로그 보존 → 진행로그 Notion 첨부로 우회).
4. **낯선 사람이 쓰고 있었다** — 외부 유입 사건 (부동산 법률 상담·서울시 공공회의록 포함). "공개 URL = 공개 API, 무료 배포 보호 범위는 직접 확인" 메시지. 민감 내용은 추상화.
5. **공유하고 싶은 실전 팁 4가지** (기존 "배운 점" 전면 리뉴얼) — 작게 검증 / 증상에서 멈추지 말기 / 실제 사용이 최고의 리뷰 / AI에게 준 규칙도 다시 묻기.

## 🎨 섹션 커버 디자인 여정 (색상 미스터리 + Codex 발견)

섹션 커버에 파트별 색상(Part 1 보라, Part 2 웜 오렌지, Part 3 그린)을 넣으려 했는데 **CSS가 계속 안 먹는 미스터리**. 시크릿 창에서도 Part 2가 보라로 표시됨. `!important`·JS 인라인 스타일 강제 주입 등 시도해도 여전히 이상. 원인 추적에 시간 상당 투자.

**Codex 리뷰가 단번에 찍음**:
> `goTo()`가 매 슬라이드 이동 시 `slides[cur].className = 'slide active'`로 className 통째 할당 → `section-cover`·`part-*` 커스텀 클래스가 **전부 날아감**. 첫 이동 후 해당 슬라이드 재방문 시 gradient 배경과 파트별 태그 색상이 모두 사라지는 구조적 버그.

**수정**: `classList.remove('active')` + `classList.toggle('prev', n > cur)` 등 classList 메서드로 전환 → 커스텀 클래스 보존.

**교훈**: "CSS가 안 먹는다"처럼 보이는 증상의 실제 원인은 JS의 className 덮어쓰기였음. **CSS·JS가 엮인 증상은 양쪽 다 봐야** — 이번 세션에서 체감한 "증상에서 멈추지 말기" 원칙의 실제 재현. 슬라이드 17번의 "실전 팁 4가지 #2" 문구가 바로 이 교훈을 담고 있는데, 그 교훈을 덱 제작 중에 또 한 번 실증한 셈.

배경 그라디언트는 사용자 선호로 제거(어두운 기본 배경 + Part 색 태그 + 큰 숫자만 남김).

## 🎓 교훈

1. **발표 자료 제작이 코드 검증 사이클을 돌린다** — 슬라이드 파이프라인 참조 박스 검증 → 서버 유의어 드리프트 발견(별도 커밋). 시각화가 invariant 체크로 자연스럽게 확장됨.
2. **Codex read-only 리뷰의 ROI** — CSS/JS 엮인 증상을 수동 디버깅에 몇십 분 쓴 뒤 Codex 백그라운드 리뷰 한 번에 진짜 원인 지목. `/codex:review` 프로토콜(CODEX-REVIEW.md)이 실전에서 비용 대비 효과 확인됨.
3. **className 덮어쓰기는 안티패턴** — 기존 `slides[cur].className = 'slide active'` 같은 패턴은 DOM 협업(CSS·JS 양쪽에서 클래스 추가)이 있는 구조에서 은근히 파괴적. `classList` 메서드가 기본이어야 함.
